Proration Plugin

PolicyCenter has the following uses for the proration plugin:
•
Generating transaction calculations
•
Rating integration
•
Calling the financialDaysBetween method to calculate a date range that rating uses
PolicyCenter standardizes proration calculations in the proration plugin. For example, the transaction calculator uses the proration plugin rather than directly implementing its own proration.
The built-in implementation of this plugin (ProrationPlugin) implements the IProrationPlugin plugin interface. If you need to customize proration, you can create your own class that implements this plugin interface.
For example, you can change how the plugin calculates the number of financial days between two dates.

Leap Days and Proration

In the base configuration, the proration plugin ignores leap days when calculating prorated premium if the IgnoreLeapDayForEffDatedCalc in config.xml is set to true. In the base configuration, this parameter applies to both prorated premiums and scalable fields.
If you need to calculate leap days differently for prorated premiums than for scalable fields, you can add an IgnoreLeapDays plugin parameter to the IProrationPlugin. The ProrationPlugin checks for this plugin parameter.
To add a plugin parameter to the proration plugin
1.
In Studio, navigate to configuration → config → Plugins → registry and open IProrationPlugin.gwp.
2.
Under Parameters, click to add a parameter.
3.
In the Name field, enter IgnoreLeapDays.
4.
In the Value field, enter true or false.
See also
•
“IgnoreLeapDayForEffDatedCalc” in the Configuration Guide.

Get Prorater for Rounding Method

Any implementation of the proration plugin interface must implement one method called getProraterForRounding. This method must return an instance of the Prorater class. The recommended way to implement this is for the plugin to declare an inner class that extends the Prorater class. In ProrationPlugin, the ForGivenRoundingLevel class extends Prorater.
Class that Extends the Prorater Class
The inner class overrides some methods of Prorater, as described in the following table. You must override the construct and the prorateFromStart methods.
 
Prorater method to override
Description
construct
Set private variables to store the rounding level and rounding mode from constructor parameters. The constructor must call super.
You must override this method.
toString
Returns a description of the proration engine.
financialDaysBetween
Calculates the number of financial days between two dates. In the base configuration, includes or ignores a leap day based on configuration parameters. See below for details.
findEndOfRatedTerm
Takes a start date and a number of days and returns the corresponding end date of the policy term. In the base configuration, includes or ignores a leap day based on configuration parameters. See below for details.
prorateFromStart
Prorates an amount of money. Its parameters are a period start date, a period end date, a date to prorate to, and an amount.
You must override this method.
scaleAmount
Scales an amount with the rounding level and rounding mode passed to the constructor.
In the base configuration, the financialDaysBetween and findEndOfRatedTerm methods ignore a leap day if:
•
The IgnoreLeapDays parameter on the plugin interface is true.
•
The IgnoreLeapDays parameter is not specified, and the IgnoreLeapDayForEffDatedCalc parameter is true in config.xml.
The financialDaysBetween and findEndOfRatedTerm methods are complementary. Therefore, a change in one method often necessitates a corresponding change in the other method. For example, in the following code, the values for newDate and date2 must be the same after running the code:
     var numDays = prorater.financialDaysBetween(date1, date2)
     var newDate = prorater.findEndOfRatedTerm(date1, numDays)
