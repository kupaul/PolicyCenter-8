Populator Plugin

PolicyCenter supports importing an externally-submitted String object and using it to populate internal objects, typically Guidewire entity instances. For example, you could use this to populate entity instances such as a PolicyPeriod with external data from XML formatted data. You do not need to use XML. You can use any text-based format, such as name/value pairs on each line, or other proprietary text formatting.
IMPORTANT If you use some format other than XML, you must properly handle escaped special characters in a safe and consistent way. Similarly, design your format to handle international characters. Always test your plugin implementation to verify that your code handles escaped characters and international characters appropriately.
The interface currently has only a single method called populatePolicyPeriod. It populates a new already-created PolicyPeriod entity instance and its subobjects from String data. At the interface level, PolicyCenter does not require that this String data be in any particular format.
The method signature is:
void populatePolicyPeriod(PolicyPeriod period, String data, String parseOptions);
The parseOptions argument is a generalized hook for you to pass parse options to your code. For example, you might pass options for the following:
•
Options to enable or disable certain types of validation on incoming XML
•
Options to write messages to a system log
•
Options to email an administrator under certain conditions
In the base configuration of this plugin, PolicyCenter uses the built-in plugin implementation class gw.webservice.pc.pc700.job.ExamplePopulatorPlugin.
This built-in implementation takes XML data created by the Guidewire XML Modeler tool, which is also known as the GX modeler. The plugin uses a built-in XML model at the class path gw.webservice.pc.pc700.gxmodel.policyperiodmodel. You can add, remove, or edit that XML model in Studio at that path.
Then the plugin calls the parse method on the model. Gosu parses the XML and creates an in-memory version of a PolicyPeriod object. However, this is not yet an entity type. This is still an XML (XSD-based) type with the following fully qualified path:
gw.webservice.pc.pc700.gxmodel.policyperiodmodel.types.complex.PolicyPeriod
Next, the default plugin calls the populatePolicyPeriod(policyPeriodObject) method on this object:
PolicyCenter adds this method with an Gosu enhancement class at the path:
gw.webservice.pc.pc700.gxmodel.PolicyPeriodModelEnhancement
Refer to that enhancement class in Studio for more information about how it handles account contacts. The basic pattern is detecting whether this contact already exists, and if not creating a new contact in the database.
In a real-world implementation, you might want to add additional logic to the import process. You can add this logic in the populator plugin implementation, or encapsulate it in enhancement classes like PolicyPeriodModelEnhancement.
For example, you might want to use these hooks to populate and validate line-specific data.
When PolicyCenter Calls The Populator Plugin
PolicyCenter calls the populatePolicyPeriod plugin method when an external system calls any of the following web service APIs:
•
IPolicyRenewalAPI.startNewRenewalaccountNumber : String, productCode : String, producerCodeId : String, policyPeriodData : String, parseOptions : String)
•
ISubmissionAPI.startSubmission(accountNumber : String, productCode : String, producerCodeId : String, policyPeriodData : String, parseOptions : String).
Also, there are deprecated versions of web service APIs that import data. They internally also call this plugin.
